## 当前阶段判断

- 当前仓库已经明显超过“脚手架 / 初始骨架”阶段，整体实现接近 `Phase 8`：
  - 公共站点已具备首页、榜单、榜单目录、Relay 详情、方法论、政策、提交、Probe。
  - 运营后台已具备总览、Relay 管理、Intake、监控 Key、Sponsor、价格维护。
  - API 已具备 `public`、`admin`、`public/probe/check`、快照刷新、seed、调度基础能力。
  - Playwright 与部分 API / shared contract 测试已经落地。
- 但按项目目标评估，当前状态更适合定义为：
  - `功能广度接近 MVP 完整`
  - `产品完成度、中文化、SEO/缓存、公共页完整度、安全硬化、测试验收基线仍需收口`

## 三类用户视角判断

### 1. 运营人员

已有后台与基础工作流，但仍偏工程控制台，主要问题：

- 中文化尚未完成，术语、状态、表单提示与操作反馈几乎都是英文。
- 运营文档不足，提交审核、Key 轮换、Sponsor 管理、刷新 public 数据的边界说明不够清晰。
- 当前后台更适合熟悉代码的维护者，不够适合中文运营团队直接接手。

### 2. 普通用户

已有可用的公共浏览路径，但信息完整度不足，主要问题：

- 首页没有把最新异常 / 事故信息模块真正展示出来。
- 榜单页缺少面向普通用户的说明性结构，例如榜单说明、自然排名与赞助位分层提示、更多筛选边界。
- Relay 详情页缺少完整的价格历史与事故时间线模块，影响“解释性”和“信任感”。

### 3. 中转站节点运营者

已有 `/submit` 和 `/probe` 入口，但仍存在关键短板：

- `/submit`、`/probe` 的用户引导几乎全英文，不符合最终中文使用场景。
- Probe 结果虽然已有一定解释性，但中文化、术语统一、错误提示分层仍需完善。
- Probe 安全实现距文档要求仍有硬化缺口，特别是 DNS rebinding / TOCTOU、应用侧限流与上线约束。

## 主要问题清单

按优先级从高到低整理如下。

### P0 - 前后台未完成中文化

现状：

- `apps/web/index.html`
- `apps/admin/index.html`
- `apps/web/src/app.tsx`
- `apps/admin/src/app.tsx`

存在问题：

- 页面语言仍为 `en`。
- 公共站点与后台的大部分导航、按钮、表单、错误提示、帮助信息、状态文案均为英文。
- 日期时间展示依赖浏览器默认 locale，没有形成 `zh-CN` 与中国用户语境的一致展示策略。

影响：

- 直接影响中国用户、运营人员、中转站节点运营者的可用性。
- 这是当前最明显、最确定的目标偏差。

### P0 - Relay 详情页未补齐文档要求的完整模块

现状：

- 文档要求 Relay 详情页包含：
  - 概览
  - 历史趋势
  - 模型支持
  - 价格历史
  - 事故时间线
- 当前实现已加载部分数据，但页面只完整呈现了概览、趋势和模型 / 当前价格信息。

存在问题：

- 缺少明确的价格历史展示模块。
- 缺少事故时间线模块。

影响：

- 普通用户无法完整理解 Relay 的稳定性与风险变化。
- 文档、API 能力、UI 实现三者开始漂移。

### P0 - Public Probe 安全实现与文档目标存在差距

现状：

- 公开 Probe 已可用，但仍需硬化。

存在问题：

- DNS 解析与真实请求连接之间仍存在 TOCTOU / rebinding 风险。
- 应用侧限流、Turnstile、连接级约束、上线边界没有完全落到实现。
- 当前更适合“受控试运行”，不适合视为已完全硬化的公网能力。

影响：

- 这是中转站节点运营者场景里最敏感的风险点。
- 若继续公开扩大使用，安全与成本风险都偏高。

### P0 - 公共 API 缓存策略未真正落地

现状：

- 架构文档要求 `/public/*` 可缓存，并显式设置 `Cache-Control` 与 `Cloudflare-CDN-Cache-Control`。
- 当前 `apps/api/src/routes/public.ts` 基本直接返回 JSON。
- `apps/api-edge/src/index.ts` 目前只是透明转发。

存在问题：

- 缺少统一的公共缓存响应头策略。
- 文档与实现不一致。

影响：

- 首页、榜单、详情页的稳定性、性能与 Cloudflare CDN 命中率都受影响。

### P1 - 首页信息完整度不足

现状：

- 首页已具备 hero、精选榜单、方法论 / policy 引导、sponsor 展示。

存在问题：

- `latestIncidents` 在 API 合同中存在，但首页未真正渲染。
- 用户缺少“最近异常 / 风险提示”入口。
- 与 `docs/ROUTES.md` 的预期结构不一致。

### P1 - 榜单页解释性不足

现状：

- 榜单页已经能正常呈现主要数据。

存在问题：

- 缺少更明确的自然排名说明。
- 缺少面向普通用户的阅读指引与限制说明。
- Sponsor 分层虽然已有理念，但榜单页上的结构化表达不够完整。

### P1 - SEO / 分享能力偏弱

现状：

- 目前仍为 CSR SPA。

存在问题：

- 缺少 route-level title、description、canonical、OG 基础策略。
- 缺少结构化数据。
- 没有首页 / 榜单 / Relay 详情页的预渲染或 SSR 过渡方案。

影响：

- 不利于搜索获取和分享卡片质量。
- 与“公共页面在 SEO 有收益时优先 SSR / pre-render”的产品规则存在偏差。

### P1 - public / admin 代码边界需要清理

现状：

- `/public/submissions` 当前位于 `apps/api/src/routes/admin.ts`
- 对应 schema 也和 admin schema 混放在同一个 shared 文件中。

存在问题：

- 路径前缀没错，但代码组织已经让 public/admin 责任边界变得模糊。
- 不利于后续补 internal 层与进一步扩展。

### P2 - 测试验收基线需要切换到中文目标

现状：

- 现有 Playwright 覆盖较好，但断言几乎全部是英文 happy path。

存在问题：

- 一旦前后台全面中文化，现有测试需要整体迁移。
- 当前缺少对缓存头、SEO meta、中文文案与中文时间展示的检查。

## 结论与优先级建议

后续自动迭代建议以以下优先级推进：

1. 完成前后台中文化与中国用户展示规则统一。
2. 修复首页、榜单、Relay 详情页的信息完整度缺口。
3. 补齐公共 API 缓存头与 public / admin 代码边界。
4. 增强 SEO 基础能力与页面级 metadata。
5. 将 Playwright 与 API 验证基线切换为中文产品目标。

## 复审追加发现（第 5 轮补充）

在第 2-4 轮完成后，又进行了一轮子代理复审与本地验收，额外发现以下问题，并已在代码中收口：

- `apps/api/src/app.ts`
  - `/public/*` 的缓存头原先会覆盖到 `>= 400` 的错误响应。
  - 已调整为仅对成功的公共 `GET` 响应注入缓存头，避免错误响应被 CDN 误标记为可缓存。
- `apps/admin/src/app.tsx`
  - 中文日期时间虽然已经本地化，但仍受浏览器时区影响。
  - 已固定为 `Asia/Shanghai`，让中国运营团队看到的时间口径一致。
- `apps/web/src/app.tsx`
  - Relay 详情页的历史趋势、状态、价格历史、事故时间线模块在请求失败时会停留在 loading 文案。
  - 已补齐错误态展示，避免用户误解为页面仍在正常加载。
  - Relay 详情预取中的事故时间线 key 与真实请求参数不一致，已统一为 `?window=30d`。
- `e2e/public.spec.ts`
  - 中文化后的评分弹层、移动端榜单字段文案与旧断言存在偏差。
  - 已改为对齐中文产品文案，避免“Breakdown”“Avail 24h”这一类旧英文断言继续误报。
- `e2e/admin.spec.ts`
  - 原有个别断言依赖不存在的英文 label 或对成功提示使用 `exact: true`，抗文案微调能力较弱。
  - 已改为更稳定的正向字段断言与较宽松的成功提示匹配。
- `playwright.config.ts`、`scripts/e2e-start-api.sh`
  - 本地 E2E 曾受 ffmpeg 缺失、Playwright 浏览器下载受阻、Postgres 刚启动时迁移过早执行等环境问题影响。
  - 已补充本机 Chrome 回退、默认关闭视频录制、数据库迁移 / seed 重试逻辑，提高本地全自动验收稳定性。

这轮复审说明：当前 MVP 的主要风险已经从“功能缺失”转向“中文产品细节、错误态与验收稳定性收口”。
