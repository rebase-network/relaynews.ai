# 管理后台操作手册

## 适用范围

本文面向 `relaynew.ai` 的中文运营人员，说明当前 `admin.relaynew.ai` 后台已经实现的日常操作流程、页面职责、注意事项和已知限制。

当前后台是运营控制台，不负责：

- 前台站点发布
- Cloudflare / GitHub 部署操作
- API 服务部署
- 数据库手工维护

这些事项仍应按 `docs/DEPLOYMENT.md` 处理，不能从后台页面直接完成。

## 进入后台前的准备

### 访问地址

- 管理后台：`https://admin.relaynew.ai`
- 后端 API：`https://api.relaynew.ai`

对应路由说明见 `docs/ROUTES.md`。

### 登录方式

生产环境默认通过 API Basic Auth 保护 `/admin/*` 接口，前端会先要求输入管理员用户名和密码，再请求 `GET /admin/overview` 验证身份。

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/app.ts`
- `apps/api/src/lib/admin-auth.ts`

注意事项：

- 登录态保存在浏览器 `sessionStorage`，刷新页面通常不会丢失；退出登录或关闭当前浏览器会话后需要重新登录。
- 生产环境应始终开启管理员认证；相关配置见 `docs/DEPLOYMENT.md`。

### 时间口径

后台中的时间展示统一固定为 `Asia/Shanghai`，方便中国运营团队按同一时区判断数据。

实现参考：

- `apps/admin/src/app.tsx`

## 日常运营建议顺序

建议按下面顺序完成每天的后台处理：

1. 先看 `概览`，确认待审核提交数、赞助位数量和价格记录规模。
2. 优先处理 `审核队列`，因为新提交会直接影响目录收录和后续监测。
3. 进入 `中转站`，检查已批准 Relay 是否缺少监测密钥、状态是否正确。
4. 如需换 Key、修复监测或排查失败，进入 `密钥`。
5. 如有商务合作，再进入 `赞助位`。
6. 如需新增模型、停用旧模型或修正价格单位，进入 `模型`。
7. 如价格有变动，再进入 `价格` 追加新价格记录。

当前后台导航与页面实现见：

- `apps/admin/src/app.tsx`

## 页面职责总览

| 页面 | 路由 | 当前用途 | 主要接口 |
|---|---|---|---|
| 概览 | `/` | 查看核心统计与推荐操作顺序 | `GET /admin/overview` |
| 中转站 | `/relays` | 创建、编辑、归档 Relay；跳转到密钥页或前台详情页 | `GET/POST/PATCH/DELETE /admin/relays` |
| 审核队列 | `/intake` | 处理用户提交，批准后自动交接到 Relay | `GET /admin/submissions`、`POST /admin/submissions/:id/review` |
| 密钥 | `/credentials` | 绑定 Relay 监测 Key、重跑 Probe、轮换、删除 | `GET /admin/probe-credentials*` |
| 赞助位 | `/sponsors` | 查看、新增、编辑、删除赞助位计划 | `GET/POST/PATCH/DELETE /admin/sponsors` |
| 模型 | `/models` | 维护模型目录、价格单位与启停状态 | `GET/POST/PATCH /admin/models` |
| 价格 | `/prices` | 查看、新增、编辑、删除价格记录 | `GET /admin/prices`、`GET /admin/models`、`POST/PATCH/DELETE /admin/prices` |

接口边界补充见 `docs/INTERNAL_API_NOTES.md`。

## 页面操作说明

### 1. 概览

用途：

- 查看 `中转站总数`
- 查看 `待审核提交`
- 查看 `投放中赞助位`
- 查看 `价格记录`

页面还会给出推荐工作流入口：

- 先看审核队列
- 批准并启用
- 后续运营维护

适用场景：

- 每日值班时快速判断今天是否有新增提交
- 确认运营工作重点是否应该先处理审核还是先维护现有 Relay

注意：

- 该页除了统计和跳转，还提供“手动刷新公开快照”入口。
- 审核、价格和赞助位写操作通常会自动刷新；只有在需要人工确认同步结果时，才建议手动触发一次。

### 2. 审核队列

用途：

- 查看所有提交记录，按创建时间倒序展示
- 优先处理 `待审核`
- 查看初始 Probe 快照、测试模型、Key 预览、最近状态和错误消息
- 填写审核备注
- 对单条记录执行：
  - `批准并启用`
  - `拒绝`
  - `归档`

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/routes/admin.ts`
- `packages/shared/src/admin.ts`

#### 批准并启用会发生什么

点击 `批准并启用` 后，系统会一次完成以下动作：

1. 创建或关联对应 Relay。
2. 将提交记录状态更新为 `approved`。
3. 将提交时的活跃测试 Key 从 `submission_id` 归属迁移到 `relay_id`。
4. 如果该 Relay 之前已有活跃监测 Key，旧 Key 会先标记为 `rotated`。
5. 立即触发一次 Relay 自有监测。
6. 刷新公开快照数据。

这条链路是当前推荐的标准收录流程，不需要先去 `中转站` 页面手动创建 Relay。

#### 拒绝或归档会发生什么

- 提交状态会变为 `rejected` 或 `archived`
- 该提交关联的活跃测试 Key 会被标记为 `revoked`

#### 审核建议

- 如果初始 Probe 已失败，不代表一定要拒绝；应结合站点说明、测试模型、错误消息和提交来源判断。
- 如果确认站点可用但测试 Key 失效，建议先拒绝当前提交，让站点方重新提交可用 Key；当前 UI 不支持在审核队列内直接替换提交 Key。
- 批准后若发现仍无监测数据，下一步应去 `中转站` 或 `密钥` 页面确认活跃 Key 是否已经接管成功。

### 3. 中转站

用途：

- 查看当前非归档 Relay 列表
- 查看每个 Relay 是否已有监测 Key
- 创建新 Relay
- 编辑现有 Relay
- 归档 Relay
- 跳转到：
  - `管理密钥`
  - `前台详情页`

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/routes/admin.ts`

#### 当前 UI 可编辑字段

当前表单支持编辑：

- `标识 Slug`
- `名称`
- `基础 URL`
- `提供方`
- `官网地址`
- `文档地址`
- `中转站介绍`
- `内部备注`
- `目录状态`
- `设为精选`
- `显示赞助提示`

#### 归档的含义

点击 `归档` 不会物理删除数据库记录，而是：

- 将 Relay 标记为 `archived`
- 从运营列表中隐藏
- 取消精选与赞助提示
- 触发公开快照刷新

因此它更接近“从当前运营视图下线”，不是永久删除。

#### 运营建议

- 审核通过后，先在这里确认 Relay 已经进入目录。
- 如果卡片上显示“缺少监测密钥”，应立即进入 `管理密钥` 补绑。
- 如果站点临时停服，可先把 `目录状态` 调整为 `已暂停`；如果确定不再维护，再考虑 `已退役` 或 `归档`。

### 4. 密钥

用途：

- 只展示已批准 Relay 的监测密钥
- 新增监测 Key
- 查看密钥详情
- 显示 / 复制密钥
- `重新运行 Probe`
- `撤销密钥`
- `轮换密钥`
- `删除密钥`

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/routes/admin.ts`

#### 这里不会显示什么

待审核提交里的测试 Key 不会出现在本页，它们会保留在 `审核队列` 中。只有 Relay 自有监测 Key 会进入这里。

#### 推荐使用方式

- 新批准的 Relay 如果还没有 Key，用 `绑定监测密钥`
- 已有 Key 但怀疑状态过期，用 `重新运行 Probe`
- 如果想停用一条密钥但保留历史记录，用 `撤销密钥`
- 需要替换 Key，用 `轮换密钥`
- 确认某条记录应彻底移除时，再用 `删除密钥`

#### 撤销、轮换和删除的区别

- `撤销密钥`
  - 会把当前密钥标记为 `revoked`
  - 记录会继续保留
  - 适合先停用、后续还要追踪历史的情况

- `轮换密钥`
  - 会创建新的活跃 Key
  - 旧 Key 会变成 `rotated`
  - 新 Key 会立即触发一次 Probe
- `删除密钥`
  - 会直接删除整条记录
  - 更适合误录入或确认不再需要保留时使用

如果只是临时停用但还想保留记录，优先考虑撤销；如果是替换为新 Key，优先考虑轮换；都不建议直接删除后再人工补记。

#### 当前限制

- 当前页面也没有批量轮换、批量重跑、搜索或筛选能力。

### 5. 赞助位

用途：

- 查看已有赞助位列表
- 新建赞助位
- 编辑已有赞助位
- 选择是否关联某个 Relay
- 设置投放位标识、状态和时间窗口

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/routes/admin.ts`

运营原则：

- 赞助展示与自然排名必须分离。
- 赞助位不会直接修改榜单自然排序。

推荐做法：

- `placement` 使用稳定、可复用的投放位标识
- 时间一律填写 ISO 时间字符串，避免模糊时区
- 如果只是商务占位但尚未上线，可先设为 `草稿`
- 如果只是想暂停投放，不必重建记录，直接编辑状态为 `已暂停`
- 如果合作结束，优先把状态改为 `已结束`，保留历史记录

当前可做操作：

- 创建赞助位
- 编辑名称、投放位标识、关联 Relay、状态、开始时间、结束时间
- 删除误建或重复的赞助位记录

当前限制：

- 当前 UI 也没有批量调整多个赞助位的能力。

### 6. 模型

用途：

- 查看模型目录与启停状态
- 新增模型
- 编辑模型键值、名称、提供方、分类和价格单位
- 控制模型是否可用于价格录入

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/routes/admin.ts`

推荐做法：

- `key` 使用稳定的英文标识，避免频繁变更。
- `name` 使用前台希望展示的模型名称。
- `family` 用于站点目录和榜单聚合分类，尽量保持同系列模型一致。
- 若只是暂时不想继续录入价格，优先把模型改为 `停用`，不要直接清理历史数据。
- 调整价格单位时，只需要编辑模型本身，无需回填旧价格记录。

当前限制：

- 当前 UI 不提供删除模型；如需彻底移除模型并清理关联数据，仍需工程确认依赖关系后处理。

### 7. 价格

用途：

- 查看已有价格历史
- 选择 Relay 与模型
- 录入或修正输入价、输出价、生效时间、货币和来源
- 追加新价格记录
- 编辑既有价格记录
- 删除误录的价格记录

实现参考：

- `apps/admin/src/app.tsx`
- `apps/api/src/routes/admin.ts`

推荐做法：

- 把价格记录当作“价格历史追加”，不要把它当作覆盖式修改。
- 价格变化时，新增一条新的 `effectiveFrom` 记录即可。
- 至少要填写输入价或输出价中的一个。
- 只有在确认是误录、重复或需要直接修正旧数据时，才建议编辑或删除。

当前限制：

- 价格页默认只展示“启用中”的模型；若要继续录入某个已停用模型，需要先回到 `模型` 页面重新启用。

## 典型操作流程

### 场景 1：处理新的站点提交

1. 打开 `审核队列`
2. 查看站点名称、Base URL、Relay 说明、测试模型和初始 Probe 快照
3. 在备注框写入审核结论
4. 点击 `批准并启用`
5. 跳转到 `中转站` 确认记录已出现
6. 如果出现“缺少监测密钥”，点击 `管理密钥` 继续处理

### 场景 2：Relay 已收录，但监测异常

1. 打开 `中转站`，找到对应 Relay
2. 点击 `管理密钥`
3. 在 `监测密钥详情` 中先看最近一次 Probe 状态、HTTP 状态码和实际探测地址
4. 先尝试 `重新运行 Probe`
5. 如果仍失败，再执行 `轮换密钥`
6. 若问题仍无法定位，交由工程排查协议兼容、网络或后端逻辑

### 场景 3：站点停止运营或不再公开展示

1. 打开 `中转站`
2. 视情况将目录状态改为：
   - `已暂停`
   - `已退役`
   - 或直接 `归档`
3. 若不再需要监测，可再到 `密钥` 页面删除对应 Key

### 场景 4：新增或停用模型

1. 打开 `模型`
2. 如果是新增模型，填写 `key`、名称、提供方、分类和价格单位后点击创建
3. 如果是不再继续录入价格的模型，编辑后取消“在价格录入中启用”
4. 保存后，再去 `价格` 页面确认下拉框是否符合预期

### 场景 5：价格调整

1. 打开 `价格`
2. 选择 Relay 和模型
3. 填入新的输入 / 输出价格
4. 填写新的 `生效时间`
5. 创建新记录，不改旧记录
