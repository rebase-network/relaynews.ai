# 管理后台操作手册

## 适用范围

本文面向 `relaynew.ai` 的中文运营人员，说明当前 `a.relaynew.ai` 后台已经实现的日常操作流程、页面职责、注意事项和当前限制。

当前后台是运营控制台，不负责：

- 前台站点发布
- Cloudflare / GitHub 部署操作
- API 服务部署
- 数据库手工维护

这些事项仍应按 `docs/TECHNICAL_OPERATIONS.md` 处理，不能从后台页面直接完成。

## 进入后台前的准备

### 访问地址

- 管理后台：`https://a.relaynew.ai`
- 后端 API：`https://api.relaynew.ai`

对应路由说明见 `docs/ROUTES.md`。

### 登录方式

生产环境默认通过 API Basic Auth 保护 `/admin/*` 接口，前端会先要求输入管理员用户名和密码，再请求 `GET /admin/overview` 验证身份。

注意事项：

- 登录态保存在浏览器 `sessionStorage`，刷新页面通常不会丢失；退出登录或关闭当前浏览器会话后需要重新登录。
- 生产环境应始终开启管理员认证；相关配置见 `docs/TECHNICAL_OPERATIONS.md`。

### 时间口径

后台中的时间展示统一固定为 `Asia/Shanghai`，方便中国运营团队按同一时区判断数据。

## 当前运营原则

当前后台已经收敛为两条主线：

1. `提交记录`
   - 这里只处理待审核的新提交。
   - 一旦审核完成，记录就会离开当前队列。
2. `Relay`
   - 这里是长期维护的主工作台。
   - 审核通过后的站点会进入 Relay 列表，由运营继续维护资料、状态、Key 和价格信息。

配套规则如下：

- `approved` 的提交记录会进入 `提交历史`，同时创建或关联 Relay。
- `rejected` 与 `archived` 的提交记录也只保留在 `提交历史`。
- 只有 `active` 状态的 Relay 会参与自动测试、公开目录和榜单展示。
- `paused` 状态的 Relay 仍可编辑，但不会参与测试，也不会出现在目录和榜单里。
- `archived` 状态的 Relay 会离开当前 Relay 列表，只在 `Relay历史` 中展示；后续可重新激活。
- API Key 的新增、轮换和替换属于管理员操作，但日常入口应围绕 Relay 编辑完成，而不是先从独立密钥中心开始。
- 管理后台没有单独的可见“概览页”；实际默认进入的是 `Relay` 列表，`GET /admin/overview` 主要用于登录校验与后台 bootstrap。

## 建议的日常处理顺序

建议按下面顺序完成每天的后台处理：

1. 先进入 `提交记录`，优先处理所有 `pending` 记录。
2. 对已通过审核的站点，转到 `Relay` 列表继续完善资料、价格表、站点状态和测试 Key。
3. 如需查看过往审批结果，使用 `提交历史`。
4. 如需处理下线站点、恢复旧站点或查看归档 Relay，使用 `Relay历史`。
5. 如有商务合作，再处理 `赞助位`。
6. 如需维护模型目录、价格单位或开关，进入 `模型`。
7. 只有在做补充排查时，再通过直达 URL 进入 `密钥` 或 `价格` 页面。

## 页面职责总览

### 主导航页面

| 页面 | 路由 | 当前用途 | 主要接口 |
|---|---|---|---|
| Relay | `/relays` | 默认入口；查看 active / paused Relay，手动新增、用右侧抽屉查看详情、编辑、暂停、归档、重新激活 | `GET/POST/PATCH/DELETE /admin/relays` |
| Relay历史 | `/relays/history` | 查看 archived Relay，用右侧抽屉回看详情、编辑或重新激活 | `GET /admin/relays` |
| 提交记录 | `/intake` | 处理当前待审核提交 | `GET /admin/submissions`、`POST /admin/submissions/:id/review` |
| 提交历史 | `/intake/history` | 查看 approved / rejected / archived 的提交记录，并在右侧抽屉回看原始资料与审核结果 | `GET /admin/submissions` |
| 赞助位 | `/sponsors` | 从 active Relay 中挑选前台赞助展示条目，并支持移除；当前页面不再暴露 placement / 时间窗口编辑 | `GET/POST/DELETE /admin/sponsors` |
| 模型 | `/models` | 维护模型键值、价格单位与启停状态，并支持删除；厂商与分类由模型键值自动推导 | `GET/POST/PATCH/DELETE /admin/models` |

### 次级工具页面

这些页面仍然存在，但当前实现里不在主导航中，通常只在补充运维或历史排查时直接访问：

| 页面 | 路由 | 当前用途 | 主要接口 |
|---|---|---|---|
| 密钥 | `/credentials` | 直接查看或排查已有监测 Key，做轮换、撤销、重跑 | `GET /admin/probe-credentials*` |
| 价格 | `/prices` | 直接维护价格记录的兼容入口 | `GET /admin/prices`、`POST/PATCH/DELETE /admin/prices` |

## 关键流程说明

### 1. 提交记录

用途：

- 查看所有 `pending` 提交
- 审阅提交资料、站点说明、模型价格表和初始测试结果
- 填写审核备注
- 对单条记录执行：
  - `批准并启用`
  - `拒绝`
  - `归档`

当前提交表单会包含以下字段：

- 站点名字
- 站点网站
- 联系方式
- 站点简介
- `Base URL`
- 测试 `API Key`
- 支持的模型及价格表

模型价格表中每行包含：

- 模型
- Input 价格
- Output 价格

#### 批准并启用后会发生什么

点击 `批准并启用` 后，系统会自动完成以下动作：

1. 创建或关联 Relay。
2. 把提交中的站点资料写入 Relay，包括名称、网站、联系方式、简介、`Base URL`。
3. 把提交中的模型价格表同步到 Relay 目录数据。
4. 将提交关联的活跃测试 Key 转移到对应 Relay。
5. 将 Relay 状态置为 `active`。
6. 立即触发一次 Relay 自有测试，并刷新公开快照。
7. 把这条提交从当前队列移到 `提交历史`。

#### 拒绝或归档后会发生什么

- 提交状态会变为 `rejected` 或 `archived`
- 当前提交不再出现在 `提交记录`
- 记录仅保留在 `提交历史`
- 如果提交关联了活跃测试 Key，它会被撤销，不会继续用于监测

#### 审核建议

- 初始测试失败并不一定代表必须拒绝，应结合站点说明、模型价格表和错误信息综合判断。
- 如果只是测试 Key 已失效，建议拒绝当前提交，让站点方重新提交有效 Key。
- 审核通过后，下一步应进入 `Relay` 页面确认资料完整度和 Relay 状态是否正确。

### 2. Relay

用途：

- 查看当前 `active` 与 `paused` 的 Relay 列表
- 手动新增 Relay
- 在右侧抽屉查看现有 Relay 详情
- 在右侧抽屉编辑现有 Relay
- 将 Relay 暂停或归档
- 将 `paused` Relay 重新激活
- 在需要时查看 Relay 的当前监测 Key 摘要与模型价格表

当前 Relay 信息应包含：

- 站点名字
- 站点网站
- 联系方式
- 站点简介
- `Base URL`
- 测试 `API Key`
- 支持的模型及价格表

#### Relay 状态的含义

- `active`
  - 参与自动测试
  - 可进入公开目录和榜单
- `paused`
  - 不参与自动测试
  - 不进入公开目录和榜单
  - 但仍可编辑，且可重新激活
- `archived`
  - 不在当前 Relay 列表中展示
  - 只在 `Relay历史` 中展示
  - 可在需要时重新激活

#### 运营建议

- 审核通过后，应第一时间在这里检查 Relay 资料是否完整。
- 如果站点只是临时维护，优先使用 `paused`，不要直接 `archived`。
- 如果站点确实已经不再对外提供服务，再考虑归档。
- 价格信息的日常维护优先在 Relay 编辑表单里完成，而不是先去旧的价格页。
- 当前手动新增 Relay 时，需要同时提供至少一行模型价格和一个测试 Key，这样创建完成后才能直接进入现有测试流程。

### 3. Relay历史

用途：

- 查看所有已归档 Relay
- 回看历史资料
- 在需要时重新激活站点

适用场景：

- 某个站点曾经下线，后续重新恢复
- 需要核对曾经归档过的历史站点资料

### 4. 提交历史

用途：

- 查看所有已审核完成的提交
- 回看提交时提供的原始说明、模型价格表和初始测试结果
- 核对某个 Relay 最初是如何进入目录的

适用场景：

- 追踪某个 Relay 的来源
- 复盘一次审核结论为什么是通过、拒绝或归档
- 对比提交时资料与当前 Relay 资料的差异

### 5. 密钥（次级工具）

当前页面仍可直接访问，但不再是主工作流入口，也不在主导航中。

适用场景：

- 统一排查已有监测 Key 的状态
- 直接查看某条 Key 的详细探测结果
- 做批量重跑、撤销、轮换等补充运维动作

使用建议：

- 日常新增或替换 Key，优先通过 Relay 编辑操作完成。
- 只有在做批量排查、历史核对或兼容性问题分析时，才建议直接进入该页。

### 6. 价格（次级工具）

当前页面仍可直接访问，但已经不是 Relay 价格维护的主入口。

使用建议：

- 日常维护站点价格时，优先在 Relay 编辑中使用模型价格表。
- 只有在需要直接补录、修正某条历史价格记录时，再进入该页面。

### 7. 赞助位

用途：

- 查看当前前台赞助展示列表
- 从 `active` Relay 中选择一个加入赞助展示
- 将某个 Relay 从赞助展示中移除

注意：

- 赞助位展示与自然排名严格分离。
- 赞助不会改写榜单结果，也不会影响自动化测试分数。
- 当前后台页面不再让运营手工填写 `placement`、`status`、`startAt`、`endAt`；这些值由系统按固定规则写入后台记录。
- 只有 `active` Relay 才能被加入赞助展示列表。

### 8. 模型

用途：

- 维护模型目录
- 设置价格单位
- 启用或停用某个模型
- 删除不再需要的模型键

适用场景：

- 新增一个站点价格表中需要引用的新模型
- 停用已经不再维护的模型键
- 修正模型键值或价格单位

当前实现说明：

- 模型键值是主输入。
- 厂商与模型分类会根据键值自动推导，不需要单独维护。

## 当前限制与注意事项

### 1. 提交通过后不再留在当前队列

这是设计目标，不是异常：

- 当前待处理的只保留 `pending`
- 审核完成后的都应进入历史

如果某条记录审核完成后仍留在 `提交记录`，应视为异常，需要排查。

### 2. 只有 active Relay 才有公开影响

请始终记住：

- 进入目录
- 进入榜单
- 参与自动测试

都以 `active` 为前提。

### 3. 提交资料和 Relay 资料可能会逐步分化

提交记录保留的是“原始提交快照”，而 Relay 中维护的是“当前运营中的正式资料”。

因此：

- 看到两边文案不同不一定是错误
- 需要分清是在看原始提交，还是在看当前正式站点信息

### 4. 密钥与价格页属于兼容入口

当前后台仍保留独立的 `密钥` 与 `价格` 页面，是为了兼容直接运维和历史排查；但主流程已经转向：

- `提交记录` -> `Relay`

后续若继续精简后台，这两个页面仍有可能进一步弱化，但目前实现仍完整保留对应 API 和页面。

## 建议的值班检查清单

每天值班时，建议至少确认：

- `提交记录` 中是否有待处理新站点
- 新批准的 Relay 是否已经是 `active`
- 重要 Relay 是否被误设为 `paused` 或 `archived`
- 公开榜单和目录是否只出现 `active` Relay
- 赞助展示中的 Relay 是否仍然是正确的 active 站点
- 模型目录是否覆盖当前新提交里出现的模型键
